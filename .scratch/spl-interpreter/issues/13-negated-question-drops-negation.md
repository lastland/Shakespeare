# Negated question drops the negation on a non-pronoun left operand

Status: resolved

## Parent

Follow-up to [03](03-richer-comparatives.md) (richer comparatives). Found in the Phase 2 code review.

## What to build

A negated question (`Is X not worse than Y?`) only behaves correctly when its left value is a
pronoun. When the left value is a **character** or a **noun phrase**, the greedy `constant` rule
absorbs the lowercase `not` as just another noun-phrase word, the Earley ambiguity resolves to the
**un-negated** tree, and the negation is silently dropped — the program then dies with a misleading
"unknown noun" analysis error.

The reference grammar has no negated questions at all, so our negated question is a local superset.
This is a **decision**:

- **(a) Keep & fix** — restructure so `not` cannot be swallowed by `constant` (e.g. let the `NOT`
  comparator win over a bare `WORD` here), so negated questions work for every operand kind; or
- **(b) Drop** negated questions entirely to match the reference.

Pick one and record it (an ADR if (a) keeps the superset, alongside the other conformance decisions).

Repro (verified): `Is Romeo not worse than Juliet?` parses to
`Question(left=Constant(('Romeo','not')), comparison='lt', negated=False)`, then analysis raises
`AnalysisError: unknown noun: 'Romeo not'`. The pronoun form `Are you not worse than Juliet?`
correctly yields `negated=True`, which is why the existing tests (all pronoun-first) pass.

## Decision

b

## Agent Brief

**Category:** bug
**Decision:** (b) Drop negated questions entirely (match the reference).
**Summary:** Remove the negated-comparison form from questions so `not` is no longer accepted inside a question; this eliminates the bug where negation is silently dropped on a non-pronoun left operand.

**Current behavior:**
The grammar accepts `Is X not <comparative> Y?` via a negated alternative of the `comparison` rule
(`NOT comp_kind`), producing a `Question` with `negated=True`. This works only when the left operand
is a pronoun; with a character or noun-phrase left operand the greedy `constant` rule swallows the
`not` as a noun-phrase word, the ambiguity resolves to the un-negated tree, negation is silently
dropped, and analysis raises a misleading "unknown noun" error. The reference grammar has no negated
questions at all.

**Desired behavior:**
- A question containing `not` no longer parses — it raises a parse error — regardless of operand
  kind. Examples that must now fail to parse: `Is Romeo not worse than Juliet?`,
  `Are you not as good as me?`, `Am I not worse than you?`, `Am I not more rotten than you?`.
- Plain (non-negated) comparisons are completely unaffected: `Is X worse than Y?`,
  `Is X better than Y?`, `Is X as good as Y?`, the bare comparatives, and `more <adjective> than`
  all parse and evaluate exactly as before.

**Key interfaces:**
- The grammar's `comparison` rule should no longer offer a negated alternative — a comparison is
  just its `comp_kind`.
- The `Question` AST node currently carries a `negated: bool` field; with negation removed it is
  obsolete. Remove it cleanly from the AST and from every layer that constructs or consumes it
  (transformer, analyzer fold, interpreter comparison evaluation), so the AST contract reflects that
  negation no longer exists. The interpreter's comparison step drops the "invert if negated" logic.
- The transformer's negated/positive comparison wrappers collapse to a single un-negated path.

**Acceptance criteria:**
- [ ] A question containing `not` raises a `ParseError` for BOTH a pronoun operand and a character/noun-phrase operand; a test asserts this.
- [ ] All non-negated comparison forms still parse and evaluate correctly (regression coverage).
- [ ] The four existing negated-question tests are removed (the `not better than` interpreter case and the three `… not …` parser cases).
- [ ] `Question` no longer exposes a `negated` field (removed across ast/transformer/analyzer/interpreter).
- [ ] Full suite + differential suite (8/8 byte-exact) stay green. (No reference program uses a negated question, so the differential set is unaffected.)

**Out of scope — DO NOT TOUCH:**
- The `If not, …` conditional (`IF NOT`) is a SEPARATE construct and must keep working. The `NOT`
  terminal stays in the grammar for it — only the question's negated-comparison alternative is removed.
- The `more <neutral adjective>` rejection (#15) — separate issue, even though it touches the same comparison area.

## Resolution

Removed the negated-comparison form end to end: the grammar's `comparison` is now just its
`comp_kind` (the `NOT comp_kind` alternative and the `cmp_positive`/`cmp_negated` aliases are gone),
and the `Question.negated` field is removed across the AST, transformer, analyzer fold, and the
interpreter's `_compare` (the "invert if negated" step and the `negated` parameter). Dropping the
grammar alternative was necessary but not sufficient — the generic `WORD` terminal still matched
`not`, so the greedy `constant` rule swallowed it (`Romeo not`) and the question parsed un-negated
(the silent-drop bug, now without negation even being expressible). Fixed structurally by reserving
`not` out of `WORD` via a leading negative lookahead `(?!not(?![a-z]))` (keyword reservation per
ADR-0002): `not` is never SPL vocabulary, and words merely containing "not" (`nothing`, `another`)
are unaffected by the word boundary. A `not`-bearing question now raises `ParseError` for every
operand kind (pronoun, character, noun-phrase); the four old negated tests were deleted and a
parametrized `ParseError` test added. The separate `If not, …` conditional keeps working (the `NOT`
terminal stays). Decision recorded in ADR-0004. Verified: ruff clean, pyright 0 errors, full pytest
green (169 passed), differential 8/8 byte-exact.
