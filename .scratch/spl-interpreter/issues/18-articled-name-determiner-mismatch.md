# Articled-name retry mis-resolves a non-`the` determiner (or bare noun) to a `The X` character

Status: resolved

## Parent

Follow-up to [12](12-articled-character-names.md) (articled character names). Found in the Phase 2 code review.

## What to build

In value position the `constant` rule drops **every** determiner (`a`, `an`, `the`, `my`, `his`,
…), then the analyzer blindly re-prepends `the` to retry the character match. Because the original
determiner is unrecoverable, a non-`the` determiner — or no determiner at all — in front of a word
that matches a declared `The X` character resolves to that **character**.

Repro (verified): with `The Ghost` declared, the values `his Ghost`, `a Ghost`, and bare `Ghost` all
fold to `CharacterRef("The Ghost")`.

This is **latent** today: no canonical `The X` character's suffix (`Ghost`, `Apothecary`, the Dukes,
…) is itself a noun, so there is no active collision. It's a correctness-hardening **decision**: the
determiner is currently discarded by the grammar, so a fix must preserve enough of it (in the
grammar/AST) to know whether it was `the`, then settle the intended semantics — e.g. only `the X`
resolves to a `The X` character, while `his X` / bare `X` fall through to noun/error.

## Resolution

The `constant` grammar rule now keeps a literal leading `the` as a `THE` token (its own
alternative) while every other determiner stays inlined and dropped; the transformer records this as
`Constant.leading_the`, and the analyzer's character-match retry (`the ` re-prepended) is gated on
that flag. So `the Ghost` resolves to `CharacterRef("The Ghost")`, while `his Ghost` / `a Ghost` /
bare `Ghost` skip the retry and fall through to noun-phrase resolution (an "unknown noun" error,
matching the reference). `the Romeo` still resolves via the bare match and `the King` still folds to
the noun value (the retry finds no character and falls through). Verified: ruff, pyright (0 errors),
full suite (174 passed), and the differential suite 8/8 byte-exact (incl. `primes.spl`, which uses
`the Ghost`).

## Agent Brief

**Category:** bug
**Decision:** Match the reference — resolve a value to a `The X` character ONLY when the source wrote `the X`.
**Summary:** Stop mis-resolving a non-`the` determiner (or a bare name) to a declared `The X` character.

**Current behavior:**
A value (`constant`) drops its leading determiner during parsing, then the analyzer blindly
re-prepends `the` and retries the character match. So with `The Ghost` declared, `the Ghost`,
`his Ghost`, `a Ghost`, and bare `Ghost` all fold to a reference to the character — the determiner
that was actually written is unrecoverable.

**Desired behavior (matches the `shakespearelang` reference, whose `character_name` value is the
full literal name `The Ghost`; a possessive/article only appears in noun phrases):**
- `the Ghost` (leading article literally `the`) → reference to the character `The Ghost`.
- `his Ghost` / `a Ghost` / `my Ghost` (any other determiner) and bare `Ghost` (no determiner) →
  NOT the character; they fall through to noun-phrase resolution (here → an "unknown noun" analysis
  error, since `ghost` is not a noun) — exactly as the reference rejects them.
- A value with a leading `the` whose bare form already names a character (e.g. `the Romeo` →
  `Romeo`) still resolves to that character (the `the` is harmless when the bare match succeeds).
- A leading `the` before an ordinary noun with no matching character (e.g. `the King`, no such
  character; `king` is a noun) still folds to the noun's value, unchanged.

**Key interfaces:**
- The `Constant` value node currently carries only its words; the determiner is dropped by the
  grammar so the analyzer can't tell `the X` from `X`. Carry whether the leading determiner was
  literally `the` (e.g. a boolean on `Constant`, or capture the determiner token) from the grammar
  through the transformer into the node.
- The analyzer's constant-folding — the step that, after a bare character match fails, retries with
  `the ` prepended — must gate that retry on "the leading determiner was `the`". With any other
  determiner, or none, it does not retry and falls through to the noun-phrase value.

**Acceptance criteria:**
- [ ] With `The Ghost` declared: `the Ghost` → CharacterRef("The Ghost"); `his Ghost` / `a Ghost` / bare `Ghost` → raise an "unknown noun" analysis error (not the character). Tests cover both.
- [ ] `the King` (no `The King` character; `king` is a noun) still folds to the noun value; `the Romeo` still resolves to character `Romeo` (regression coverage).
- [ ] `primes.spl` (uses `the Ghost`) stays byte-exact; full + differential suites green.

**Out of scope:**
- Stage-direction names (the `name` rule) — already handled by issues 12/20 and correct; this issue is only the value-position `constant` path.
- The leading-article set for names is already narrowed to `the` (#20); don't revisit it.
