# `more <neutral adjective> than` resolves to greater-than instead of being rejected

Status: resolved

## Parent

Follow-up to [03](03-richer-comparatives.md) (richer comparatives). Found in the Phase 2 code review.

## What to build

`more <adjective> than` resolves its direction as "negative adjective → less-than, **anything else**
→ greater-than", which lumps the ~20 NEUTRAL adjectives (`big`, `huge`, `large`, `small`, `tiny`,
`little`, …) in with the positive ones. So `more tiny than` resolves to greater-than (semantically
backwards) and `more huge than` is accepted — whereas the reference admits `more` only with a
**positive** or a **negative** adjective; a neutral adjective after `more` is a parse error in the
oracle.

This is a **conformance decision**:

- **(a) Reject** `more <neutral adjective> than` to match the reference (raise, like an unknown adjective); or
- **(b) Keep** the permissive superset, but record it as an allowed divergence and decide the neutral-adjective direction deliberately (not as an accident of the "else → gt" fallback).

Likely an ADR update alongside the other conformance decisions ([07](07-goto-target-scope.md)–[10](10-conditional-without-question.md)).

Repro (verified): `Am I more tiny than you?` resolves to `comparison='gt'`; the reference
`shakespeare run` parse-errors on `more huge than`.

## Decision

Reject

## Agent Brief

**Category:** bug
**Decision:** Reject `more <neutral adjective> than` (match the reference).
**Summary:** A `more <adjective> than` comparison whose adjective is neutral should raise, not silently resolve to greater-than.

**Current behavior:**
The analyzer resolves a `more <adjective> than` marker as: negative adjective → less-than, *any
other* adjective → greater-than. Because the vocabulary only distinguishes "adjective" from
"negative adjective", the ~20 neutral adjectives (`big`, `huge`, `large`, `small`, `tiny`, `little`,
…) fall into the greater-than branch — so `more tiny than` resolves to `gt` (backwards) and
`more huge than` is accepted. The reference admits `more` only with a *positive* or a *negative*
adjective; a neutral adjective there is a parse/analysis error.

**Desired behavior:**
- `more <positive adjective> than` → greater-than (unchanged).
- `more <negative adjective> than` → less-than (unchanged).
- `more <neutral adjective> than` → raises an analysis error, the same strict way an unknown adjective already does.
- `more <non-adjective word> than` → still raises (unknown adjective).

**Key interfaces / data:**
- The comparison-resolution path (the analyzer step that turns a `MoreComparative` marker into
  `"gt"`/`"lt"`) must distinguish *positive* from *neutral* adjectives. The current `Vocabulary`
  exposes only `is_adjective` and `is_negative_adjective`, so it cannot tell positive from neutral.
  Add the missing distinction — e.g. a `positive_adjectives` word-list + `is_positive_adjective`,
  mirroring how `negative_adjectives` already works — sourced to match the reference's
  `positive_adjective` list (the installed `shakespearelang` grammar, a dev dependency, is the
  source of truth). Resolution then becomes: negative → lt, positive → gt, otherwise raise.
- A new data file must be loaded and packaged like the other word-lists (the `source-include` globs
  already cover `data/*.txt`).

**Acceptance criteria:**
- [ ] `more <neutral adjective> than` (e.g. `more tiny than`, `more huge than`) raises a clear analysis error.
- [ ] `more <positive adjective> than` still resolves to gt; `more <negative adjective> than` still resolves to lt.
- [ ] The positive/neutral/negative classification matches the reference's three adjective lists.
- [ ] A unit test covers the neutral-rejection case.
- [ ] An ADR (or an addition to an existing conformance ADR) records the "reject neutral after `more`" decision.
- [ ] Full suite + differential suite (8/8 byte-exact) stay green.

**Out of scope:**
- The bare comparatives (`bigger`/`smaller`/`worse`/…) — their directions are fixed and correct.
- Adding `less <adjective> than` (intentionally omitted).
- The negated-question removal (#13) — separate issue.

## Resolution

Added the missing positive/neutral distinction to the vocabulary: a `positive_adjectives.txt`
data file + `Vocabulary.is_positive_adjective`, mirroring the `negative_adjectives` machinery and
sourced verbatim from the reference EBNF's `positive_adjective` production (36 words). The three
reference lists partition `adjectives.txt` exactly — positive (36) ∪ neutral (20) ∪ negative (32) =
88, pairwise disjoint — so neutral is just "an adjective that is neither positive nor negative" and
needs no list. The analyzer's `_resolve_comparison` now maps `more <negative>` → lt,
`more <positive>` → gt, and a neutral (or unknown) adjective **raises** `AnalysisError`, matching
the reference (which admits `more` only with a positive/negative adjective). Decision recorded in
ADR-0004. Verified: ruff clean, pyright 0 errors, full pytest green (169 passed), differential 8/8
byte-exact.
